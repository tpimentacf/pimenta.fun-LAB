Waiting Room

Cloudflare Waiting Room

A virtual queue that protects your origin during traffic spikes (sales, ticket drops, launches). Cloudflare admits users at a sustainable rate and holds the rest on a branded waiting page.

Configure first: Dashboard → Traffic → Waiting Room, then bind it to a host/path (e.g. this page). The status check below reads the waiting-room cookie the edge sets once you've passed through the queue.
Use at your own risk. This lab was built by TPimenta LAB for hands-on testing. Always consult the official Cloudflare documentation for accurate, up-to-date guidance.

Your waiting-room status

Detected from the __cf_waitingroom cookie (a signed JWT the edge issues after admission). Client JS can see the cookie's presence, not decode it.

How the queue works

SettingWhat it controls
Total active usersMax concurrent users allowed past the room.
New users per minuteAdmission rate — how fast queued users are let in.
Session durationHow long an admitted session stays valid without activity.
Queueing methodFIFO (first-in-first-out), Random, Passthrough, or Reject.
Queue-all toggleForce everyone into the queue (great for testing the room).
JSON responseReturn queue state as JSON for SPAs/native apps instead of HTML.

Custom waiting page

Override the default page with HTML using Handlebars-style variables:

{{waitTimeKnown}}      — is an estimate available?
{{waitTime}}           — estimated minutes remaining
{{queueIsFull}}        — room at capacity
{{lastUpdated}}        — timestamp of the estimate
{{refreshIntervalSeconds}}  — auto-refresh cadence

Tip: enable Queue-all temporarily to preview your custom page, then disable it.

Testing tips

  • Set Total active users and New users per minute to very low values to force a queue.
  • Use multiple browsers / incognito windows to simulate concurrent visitors.
  • Enable Queue-all to see the room immediately without generating load.
  • Disable the WARP client so each session looks like a distinct visitor.
