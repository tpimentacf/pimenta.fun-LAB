Edge Inspector

Cloudflare Edge Inspector

See exactly how your request reaches Cloudflare: which data center (colo) served you, your geolocation, HTTP protocol, TLS version, WARP status, and the response headers the edge adds — all read live from your own connection.

How it works: this page reads /cdn-cgi/trace (served automatically on every Cloudflare zone) and inspects same-origin response headers via the browser Headers API. No origin, Worker, or add-on is required — it works on any zone proxied through Cloudflare.
Use at your own risk. This lab was built by TPimenta LAB for hands-on testing. Always consult the official Cloudflare documentation for accurate, up-to-date guidance.

Reference — Cloudflare request headers

Headers Cloudflare adds to requests before they reach your origin. These are not readable from client JavaScript — see them live on the CF Echo page, which reflects them back from a Worker.

Request headerDescription
CF-Connecting-IPThe original client IP address. Use this at the origin instead of the proxy IP.
CF-Connecting-IPv6IPv6 form of the client address, when applicable.
True-Client-IPEnterprise equivalent of CF-Connecting-IP (same value, alternate name).
CF-RayUnique request ID (ends with the colo code). Quote this in support tickets to trace a request.
CF-IPCountryISO-3166 country of the visitor (requires IP Geolocation enabled on the zone).
CF-VisitorJSON describing the original scheme, e.g. {"scheme":"https"}.
CF-WorkerHost of the Worker that dispatched a subrequest, when applicable.
X-Forwarded-ForAppends the client IP to any existing chain of proxy addresses.
X-Forwarded-ProtoOriginal protocol (http/https) used by the client to reach the edge.
Accept-EncodingNormalised by Cloudflare to reflect what the edge will accept from the origin.

The richest source of edge data inside Cloudflare is the request.cf object available in Workers (country, colo, ASN, TLS cipher, bot score, and more). See the HTTP request headers and request.cf properties docs.
