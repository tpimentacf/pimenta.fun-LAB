Rate Limit by Worker

Enforce a limit in code with the native Workers Rate Limiting binding (env.WORKER_RL.limit()). Allowed requests reach the origin; once the threshold is crossed the Worker returns 429 — only after your code decides to check.

This lab uses 5 requests / 60 seconds, keyed on the visitor IP + path. Counters are per Cloudflare data center. The binding requires Wrangler ≥ 4.36.0.
Use at your own risk. This lab was built by TPimenta LAB for hands-on testing. Always consult the official Cloudflare documentation for accurate, up-to-date guidance.

When to use this vs a Rate Limiting rule

  • You need the limit applied only after specific logic runs (e.g. after auth, for one code path).
  • You want different limits per user class (free vs. paid) using different keys/bindings.
  • You want to key on something computed in code (decoded token, tenant ID) rather than raw request fields.

For simple, declarative limits on a path/zone, use a dashboard Rate Limiting rule instead — no Worker needed.

Worker configuration

Two files. The binding is declared in wrangler.toml; the Worker calls limit(). Full source lives at workers/rate-limit/ in the repo.

wrangler.toml
name = "rate-limit"
main = "worker.js"
compatibility_date = "2025-06-01"

# Native Rate Limiting binding (Wrangler >= 4.36.0).
# "name" is what the Worker references as env.WORKER_RL.
[[ratelimits]]
name = "WORKER_RL"
namespace_id = "1001"   # unique integer-as-string; shared id = shared counter

  [ratelimits.simple]
  limit = 5      # allowed requests per period
  period = 60    # window in seconds — must be 10 or 60

routes = [
  { pattern = "www.pimenta.fun/ratelimiting/worker-rl*", zone_name = "pimenta.fun" }
]
worker.js
const LIMIT = 5, PERIOD = 60; // keep in sync with wrangler.toml

export default {
  async fetch(request, env) {
    const url = new URL(request.url);
    const ip  = request.headers.get("CF-Connecting-IP") || "unknown";
    const key = ip + ":" + url.pathname; // lab key; prefer a user/API-key id in prod

    const { success } = await env.WORKER_RL.limit({ key });

    if (!success) {
      return new Response(
        "429 Too Many Requests — " + LIMIT + " req / " + PERIOD + "s for " + url.pathname,
        { status: 429, headers: { "retry-after": String(PERIOD), "cache-control": "no-store" } }
      );
    }
    return fetch(request); // allowed → go to origin
  }
};
The limit() call does not add network latency — counters are cached on the machine running your Worker and reconciled in the background within the colo.

Configure it in Cloudflare — step by step

  1. Check Wrangler. Run npx wrangler --version — you need 4.36.0 or newer for the Rate Limiting binding.
  2. Create the project. Put worker.js and wrangler.toml in a folder (see workers/rate-limit/).
  3. Declare the binding. The [[ratelimits]] block defines WORKER_RL with limit = 5, period = 60, and a unique namespace_id. No dashboard step is needed to create the limiter — it's provisioned from this config on deploy.
  4. Authenticate. npx wrangler login (or set a CLOUDFLARE_API_TOKEN).
  5. Deploy. npx wrangler deploy. This uploads the Worker, registers the binding, and attaches the route.
  6. Bind the route. The routes entry maps /ratelimiting/worker-rl* to the Worker. Confirm under Workers & Pages → your Worker → Settings → Domains & Routes. (Prefer the dashboard? Add the route there instead and drop it from the TOML.)
  7. Disable WARP while testing so your requests hit the zone directly, then click Burst 10 above or run the curl loop below.
  8. Observe. Bindings aren't shown in the dashboard — watch 429s via Workers Logs / Traces, or emit an Analytics Engine data point when limit() returns success: false.
verify from a shell
for i in $(seq 1 8); do
  curl -s -o /dev/null -w "%{http_code}\n" https://www.pimenta.fun/ratelimiting/worker-rl/
done

Good to know

  • Locality: the limit is enforced per Cloudflare data center, per key. A globally distributed test sees the limit independently in each colo.
  • Key choice: avoid raw IPs in production (shared by many users); prefer API keys, user IDs, or routes.
  • Shared counters: two bindings with the same namespace_id — even in different Workers — share the same counter.
  • Accuracy: permissive and eventually consistent — protective, not exact accounting.
